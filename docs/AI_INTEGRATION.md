# AI Integration Plan

## 1. 目标

AI 只负责把用户已有的视觉材料整理为结构化个人风格，不替用户决定审美。输出必须可以人工修改，并保留来源。

支持两条输入路线：

1. 例图：上传多张参考图，提取共同视觉规则。
2. 文字录入：原样保存用户写下的审美描述、Prompt、tokens、rules 或 `SKILL.md` 内容；仅在用户主动选择时交给 AI 整理。

Figma 不单独作为应用内输入路线。直接读取 Figma 的工作交给 Codex 更完整；若用户只想沉淀视觉风格，可把关键画板导出后走例图路线。

## 2. 推荐的低成本模型分工

### 例图解析：千问视觉模型优先

默认采用支持图片输入的 Qwen 视觉模型。开发阶段先用质量较高的 Plus 型号建立可靠基准，稳定后切换 Flash 型号压低成本。

需要提取：色板与颜色占比；字体类别、字重与层级；网格、边距、密度和对齐；圆角、边框、阴影与材质；图像或插画语言；组件形态与状态；动效倾向；反复出现的规则与可能误判。

### 文字录入与二次整理：千问文本复审优先

默认继续使用同一枚千问 Key：第一次调用读取图片并提取视觉证据；第二次调用只读取结构化草稿，不再接收图片，负责去除来源产品内容、整理规则并生成中英文 Prompt。这样减少供应商、Key 与故障点，也更符合个人工具的低维护目标。

DeepSeek 保留为用户可选的备用文本提供商，但不再参与默认链路。此前实测其长双语 JSON 审校连续出现空内容、无效 JSON 与截断，投入的适配成本已经超过当前 MVP 的收益。

实现中不硬编码即将废弃的旧模型别名；连接后读取模型列表。

### Kimi：作为可配置备选（暂缓）

不同渠道提供的 Kimi 型号与视觉能力可能不同。当前连接的是 Kimi Open Platform，默认型号为 `kimi-k3`；后续仍需读取模型列表并运行最小图片测试，再决定它承担视觉解析、长文本整理或备用角色。

当前 Open Platform 适配器使用 `kimi-k3`，并按该模型的接口约束固定发送 `temperature: 1`；千问和 DeepSeek 继续使用各自任务指定的温度。2026-07-20 实测遇到上游引擎过载，因此 Kimi 暂不参与默认生产链路，待服务恢复后再横测。

## 3. 当前 MVP 流水线

```text
图片
        ↓
千问视觉提取画面证据与候选视觉规则
        ↓
千问第二次只读取结构化草稿，删除来源产品内容、审校绝对化规则并生成最终 prompt
        ↓
用户审阅并保存为个人风格
```

文字输入默认原样保存；只有用户点击“让 AI 整理”时才调用设置中的文本模型，不经过视觉阶段。两段式不是重复看图：第一次千问负责观察，第二次千问只阅读文字草稿并以独立请求执行抽象与编辑，从而降低把原页面 Tab、图标、业务模块和信息架构写进最终 Prompt 的概率。

## 4. 输出 Schema

```json
{
  "nameSuggestions": [],
  "summary": "",
  "transferablePrinciples": [],
  "excludedSourceDetails": [],
  "uncertainOrInferred": [],
  "transferGuidelines": [],
  "transferCheck": [],
  "coreMood": [],
  "colorRules": [],
  "typographyRules": [],
  "layoutRules": [],
  "materialRules": [],
  "componentRules": [],
  "motionRules": [],
  "mustKeep": [],
  "avoid": [],
  "evidence": [],
  "confidence": {},
  "promptZh": "",
  "promptEn": ""
}
```

`evidence` 必须指出规则来自哪张图或哪段用户文字，避免模型把猜测写成用户风格。

第二阶段把 `excludedSourceDetails` 当作隔离清单：其中的内容只能保留在该字段用于追溯，不得重新出现在最终 Prompt、必留规则、布局规则或组件规则中。固定像素、固定导航数量和“所有组件必须”之类的绝对规则，只有在多张图中反复成立时才可保留；否则改写为关系、范围或条件建议。

文本复审不重复返回整份二十字段分析，而只返回紧凑“审校补丁”：经过改写的核心规则，以及分段的中英文 Prompt。网关随后把补丁与视觉阶段的证据、置信度、排除清单和推测字段合并，以降低 JSON 截断概率并减少重复 token。

### 风格迁移边界

例图解析默认是“可迁移风格提取”，不是界面复刻。模型必须把信息分为两层：

- 可迁移：字体层级、颜色关系、网格、密度、间距节奏、形状、边框、阴影、材质、图标画法和状态反馈。
- 必须排除：产品名、业务词、Tab 名称、导航目的地、具体图标含义、文案、数据、专属功能、页面顺序与原产品信息架构。

输出增加 `transferablePrinciples`、`excludedSourceDetails`、`uncertainOrInferred`、`transferGuidelines` 与 `transferCheck`。品牌吉祥物只可归纳为插画语法；静态截图无法证明的动效、响应式、深色模式或隐藏交互必须标为推测，不能进入必留规则。最终 Prompt 必须使用新产品占位符，并通过“应用到三个无关产品仍然成立”的迁移自检。

## 5. Figma 边界

- 应用不重复建设 Figma OAuth、文件树与节点解析。
- 需要结构化理解、修改或实现 Figma 时，用户直接把链接交给 Codex。
- 只需要沉淀视觉风格时，用户导出 1–4 张关键画板并使用例图路线。
- 这一取舍减少授权、凭据、文件范围和 API 维护成本，也避免把“能读 Figma”误解为模型会自动理解整个设计文件。

## 6. API Key 安全边界

当前版本通过 `server/gateway.js` 在本机调用模型：

- 浏览器从不直接持有供应商请求逻辑。
- 浏览器开发版的 Key 只保存在 Node 网关进程内存中，停止进程后自动清除。
- 桌面版使用 Electron `safeStorage` 调用操作系统加密能力，并将密文保存在该用户的应用数据目录；下次启动自动恢复。
- Key 不进入 localStorage、项目文件、日志或 GitHub，前端也不会读取已保存的密钥原文。
- 图片在浏览器本地缩放为最长边 1600 px，最多发送 4 张。

桌面版安全策略：

- Key 使用 Windows 提供的用户级加密能力保存；密文只属于当前 Windows 用户。
- 请求从桌面后端发出，不把 Key 暴露给前端 JavaScript。
- 仓库只提交 `.env.example`，绝不提交真实 Key。
- 设置单次图片数、尺寸、Token 和预算上限。
- 第一次连接只运行最小烟测，并显示预计消耗。
- 视觉与文本路线分别烟测，避免只知道整条链路失败却看不出是哪一段。
- 烟测使用 90 秒上限；完整多图视觉提取使用 240 秒上限，文本复审使用 180 秒上限。千问混合思考模型在结构化提取与复审任务中显式关闭思考模式，以减少延迟与无关推理消耗。

当前接口：

```text
GET    /v1/health
POST   /v1/config/keys
DELETE /v1/config/keys/:provider
POST   /v1/test
POST   /v1/analyze
POST   /v1/refine
```

`/v1/analyze` 会校验两阶段结果是否至少包含足够的规则以及非空中英文 Prompt。复审阶段优先请求严格 JSON 输出；若供应商不支持该参数则自动回退。文本复审返回空结构、无效 JSON、超时或接口错误时，前端都会保留已经完成的千问视觉草稿并显示“只重试文本复审”；`/v1/refine` 只发送该结构化草稿，不会重新上传或解析图片。

## 7. 成本控制

- 上传前在本地缩放图片，UI 截图长边通常不超过 1600–2048 px。
- 先用 2–4 张代表图，不把整个灵感库一次发送。
- 使用图像哈希缓存视觉解析结果。
- 只有图片变化时重跑视觉模型；修改名称或备注只跑文本整理。
- Prompt 生成和格式修复使用 Flash 型号。
- 每次调用记录提供商、型号、输入数量、Token 和估算费用。
