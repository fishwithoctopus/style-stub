# Decision Log

## 2026-07-20：MVP 采用本地无构建 Web 原型

**决定：** 第一版使用 HTML、CSS 与原生 JavaScript，不接入框架或桌面壳。

**原因：** 当前最大未知量是窄窗信息密度、视觉隐喻和 Lab 是否好用；无构建原型能最快验证这些问题，并保持后续封装空间。

**复审条件：** 核心流程稳定，需要置顶、贴边、托盘或开机启动时。

## 2026-07-20：采用“票根档案卡”而非纯购物小票

**决定：** 全局使用电影票根、展览票和热敏纸的混合隐喻。

**原因：** 纯购物小票适合文字但缺少例图空间。票根结构可以自然容纳顶部视觉样张，同时保留纸张、编号、虚线和收藏感。

## 2026-07-20：样张优先使用 CSS 实时渲染

**决定：** MVP 为词条提供代码生成的小型组件板，不依赖外部图片。

**原因：** 实时样张更适合展示按钮、阴影、模糊、网格和状态；同时避免版权、资源丢失和加载失败。

**例外：** 无法通过组件充分表达的历史风格或艺术流派，后续允许使用带来源的图像。

## 2026-07-20：第一版不接 AI

**决定：** 使用结构化词条、权重和规则模板生成 Lab 配方。

**原因：** 先验证组合机制与输出格式，避免模型成本和不稳定性掩盖产品问题。

**复审条件：** 需要更细致的冲突分析、命名、解释或新视觉生成时。

## 2026-07-20：公共词典与个人风格分离

**决定：** 公共词条是只读版本数据，Lab 与个人风格存放在本地。

**原因：** 避免用户实验污染基础定义，也为未来词库升级和个人数据迁移留下清晰边界。

## 2026-07-20：最终形态是可置顶的 Windows 桌面组件

**决定：** Web 版本继续承担快速原型，交互稳定后使用 Tauri 2 封装为无边框、透明背景、可调整尺寸和可切换始终置顶的 Windows 小组件。

**原因：** 用户需要把工具长期放在桌面侧边，同时又希望复用当前网页界面和未来开源。

**视觉约束：** 深色预览舞台不属于桌面窗口；最终只显示纸张票根本体与轻微 CSS 阴影，不采用突兀的系统粗边框。

## 2026-07-20：顶部快捷入口改为设置

**决定：** Lab 只保留在固定底部导航，顶部右侧改为设置入口。

**原因：** 重复 Lab 入口没有新增价值；窗口、置顶、AI 网关和预算需要一个稳定的全局入口。

## 2026-07-20：收录最小化并将整理动作放回对象附近

**决定：** 网页收录只保留网址必填和备注选填；分类新增放在分类条旁，分类调整放在词条详情，排序使用图鉴页长按拖动。

**原因：** 收录动作的主要任务是提供来源，命名、标签、初步分类和提示词应由 AI 起草；分类与排序不应堆进收录表单。

## 2026-07-20：Lab 最多四种风格且权重恒等于 100%

**决定：** Lab 上限为 4，推荐 2–3；每根滑块记录互不影响的自由份量，后台再将全部份量换算为合计 100% 的实际占比。

**原因：** 保持配方可以解释，避免五种以上风格互相稀释；同时避免三四种风格时，为维持表面总和而导致多个滑块互相牵动、难以摆到用户想要的位置。
# 2026-07-20：桌面关闭行为、吸附与许可证

决定：桌面版创建常驻系统托盘；窗口 `×` 只隐藏，托盘菜单提供真正退出。窗口位置、尺寸和左右吸附状态保存在用户应用数据目录，并在多显示器变化后校正。项目采用 MIT License，版权标注为 `Style Stub contributors`。

原因：隐藏到托盘可以避免误关窗口时中断本地网关；位置记忆与吸附符合桌面侧边工具的核心使用场景；MIT 对个人开源工具最轻量，也允许后续社区复用和商业使用。
